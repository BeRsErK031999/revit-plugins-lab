# IsoField source set contract

Версия контракта: `1.0`.

Расчётные изображения загружаются одним комплектом из четырёх файлов. Один файл соответствует одному расчётному слою; перед запуском worker плагин проверяет комплект целиком.

## Обязательные роли

| Роль | Направление | Грань плиты | Статус назначения |
| --- | --- | --- | --- |
| `As1X` | X | пользователь выбирает `верх/низ` | имя файла + растровый заголовок, доступна ручная коррекция |
| `As2X` | X | пользователь выбирает `верх/низ` | имя файла + растровый заголовок, доступна ручная коррекция |
| `As3Y` | Y | пользователь выбирает `верх/низ` | имя файла + растровый заголовок, доступна ручная коррекция |
| `As4Y` | Y | пользователь выбирает `верх/низ` | имя файла + растровый заголовок, доступна ручная коррекция |

До подтверждения соответствия `As1...As4` верхней/нижней грани интерфейс не должен подменять расчётную роль инженерной интерпретацией.

## Автоматическое назначение

- Роль определяется без учёта регистра по вхождению `As1X`, `As2X`, `As3Y` или `As4Y` в имя файла.
- Имя вида `...As1X1.png` корректно определяется как `As1X`.
- Верхние 96 пикселей изображения дополнительно проверяются узким оптическим классификатором маркеров `(As1X)`, `(As2X)`, `(As3Y)`, `(As4Y)` текущего экспорта ПК ЛИРА.
- Если имя и заголовок совпадают, строка получает подтверждение `имя + заголовок`.
- Если роль есть только в заголовке, она используется автоматически; если заголовок не распознан, допустим fallback на однозначное имя файла.
- Если имя и заголовок указывают разные роли, автоназначение блокируется до ручного выбора пользователя.
- Для нестандартного шрифта/масштаба заголовка классификатор может не сработать; это не блокирует однозначное назначение по имени файла.
- Пользователь может изменить роль каждого файла в выпадающем списке.

## Блокирующая валидация

Комплект готов, только если одновременно выполнены условия:

- выбрано ровно четыре разных файла;
- каждая обязательная роль назначена ровно один раз;
- каждый файл существует и читается как изображение;
- ширина и высота всех изображений совпадают.
- отсутствуют неразрешённые конфликты имени файла и растрового заголовка.

Для разрешения записи дополнительно требуется:

- для направления X назначен ровно один нижний и один верхний слой;
- для направления Y назначен ровно один нижний и один верхний слой;
- ни один слой не имеет значения `Unconfirmed`.

Распознавание изображений можно выполнить до назначения граней, но кнопка создания армирования остаётся заблокированной.

Разный pixel size считается признаком несовместимого экспорта или масштаба. Геометрическая проверка масштаба по контрольным точкам относится к этапу P3.

## Обработка

Настроенный CLI worker вызывается последовательно в порядке `As1X`, `As2X`, `As3Y`, `As4Y`. Результаты объединяются в один `IsoFieldRecognitionResult`:

- `id` контура получает префикс роли, например `As1X:zone-a`;
- `layerRole` заполняется ролью исходного файла;
- диагностика получает префикс `[As1X]` и не теряет связь с источником.

JSON-контракт зон остаётся отдельным совместимым входом и выбирается без изображений.

## Manifest комплекта

Команда `Сохранить manifest` создаёт файл `*.isofield-set.json` версии `1.0`. Он содержит:

- относительный путь к каждой карте;
- роль `As1X/As2X/As3Y/As4Y`;
- сохранённые `pixelWidth` и `pixelHeight`;
- SHA-256 содержимого файла;
- явную таблицу `role → direction → face`.

При загрузке пути разрешаются относительно каталога manifest. Текущие размер и SHA-256 сравниваются с сохранёнными значениями. Изменённый или отсутствующий источник блокирует обработку и показывает проблему в строке файла.

Пример структуры:

```json
{
  "schemaVersion": "1.0",
  "files": [
    {
      "path": "map_As1X1.png",
      "role": "As1X",
      "pixelWidth": 1584,
      "pixelHeight": 892,
      "sha256": "..."
    }
  ],
  "layerMappings": [
    {
      "role": "As1X",
      "direction": "X",
      "face": "Bottom"
    }
  ]
}
```
