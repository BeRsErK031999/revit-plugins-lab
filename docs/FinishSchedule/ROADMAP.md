# Ведомость отделки помещений — RoadMap

## Цель

Добавить в существующую вкладку `TrueBIM` раздел `АР` и инструмент `Ведомость отделки`, который:

1. выбирает размещённые и замкнутые помещения заданной области;
2. находит физические элементы отделки стен, полов и потолков;
3. заново определяет их принадлежность помещениям по геометрии;
4. считает относящуюся к каждому помещению площадь;
5. группирует помещения по составу отделки, но не по площади;
6. записывает одинаковые агрегированные текстовые значения во все помещения группы;
7. создаёт или безопасно обновляет обычную спецификацию категории `Помещения`.

Спецификация не выполняет расчёты. Она показывает подготовленные плагином значения и объединяет комнаты одной группы через `ScheduleDefinition.IsItemized = false`.

## Обратная связь `FS-011` от 21.07.2026

Лог `truebim (7).log` показал `54` помещения, `66` перекрытий пола и `114` потолков-кандидатов. После строгого spatial-index prefilter в расчёт попадало только `48` полов и `6` потолков, а boolean intersection помещения `2304268` с полом `1957984` завершался ошибкой Revit geometry kernel.

Исправление сохраняет исходный контракт разбиения одного элемента по нескольким помещениям:

- кандидаты ищутся отдельно по категории: с малым допуском вокруг нижней границы для пола и ограниченным вертикальным диапазоном до `2 м` выше для потолка;
- физический расчёт получает полный набор классифицированных элементов и самостоятельно применяет тот же категорийный spatial-index;
- при отказе сложного boolean intersection room-probe разбивается на простые треугольные призмы, площади которых суммируются только внутри конкретного помещения;
- верхняя строка спецификации имеет высоту `12 мм`, шрифт `3,5 мм` и bold; заголовки граф — `2,5 мм`;
- рамка, границы заголовков и вертикальные линии используют стиль `Обычные линии` с fallback на built-in medium lines, промежуточные горизонтали — `Тонкие линии`;
- revision hash оформления обновлён, поэтому уже существующая ведомость TrueBIM перестраивается при следующем формировании даже при неизменных пользовательских настройках.

Проверка текущего среза 17.07.2026:

- `dotnet format TrueBIM.sln --verify-no-changes` — пройдено;
- compile Revit 2019–2024 (`net48`) и Revit 2025 (`net8.0-windows`) — пройдено без ошибок и предупреждений;
- Revit 2026 compile не запускался: локальный `RevitAPI.dll` этой версии отсутствует;
- `dotnet test TrueBIM.sln --configuration Release --no-build` — 679 тестов пройдено;
- runtime smoke не выполнялся.

## Результаты архитектурного аудита `FS-000`

- `App.cs` создаёт одну общую вкладку `TrueBIM`, затем панели из `TrueBimRibbon.PanelNames` и кнопки из `TrueBimRibbon.Buttons`.
- Разделы `КР`, `ЭОМ` и остальные сейчас являются Ribbon-панелями. Поэтому `АР` добавляется на тот же уровень как панель существующей вкладки, а не как новая Autodesk Ribbon-вкладка.
- `TrueBimWindow` — базовый класс окон с общей оболочкой, а не контейнер вкладок. Новое окно наследуется от него и использует `TrueBimUi`, `TrueBimTheme`, `TrueBimBrushes` и `IconFactory`.
- Универсальное JSON-хранилище настроек уже есть: `JsonSettingsStorage`. Для профиля ведомости нужен отдельный адаптер с ключом `finish-schedule`.
- Основной проект автоматически включает новые `.cs`-файлы и собирается как `net48` для Revit 2019–2024 и `net8.0-windows` для Revit 2025–2026 через `RevitVersion` и реальные API DLL каждой версии.
- Локальные скрипты сборки покрывают 2022, 2025 и 2026; `build-installer.ps1` формирует общую матрицу артефактов. Отсутствующую локальную версию нельзя объявлять runtime-проверенной.
- Готового общего сервиса `SpatialElementGeometryCalculator` или разбиения площади перекрытия по помещениям нет. Геометрический движок должен быть модульным и покрываться отдельными интеграционными сценариями.
- Полезные существующие примеры: `JoinCutProcessingService` для предварительной фильтрации и геометрии, `OpeningViewMetadataService` для Extensible Storage, `JsonSettingsStorage` для профилей и `TrueBIM.App.Tests` для чистой логики.
- Первый срез не должен обращаться к Revit API в фоновых потоках и не должен менять модель.

## Моделировочный контракт V1

### Источники физических элементов

| Вид отделки | Категория | Параметр типа | Ожидаемое значение по умолчанию |
| --- | --- | --- | --- |
| Стены | `OST_Walls` | `Группа модели` | `Внутренняя отделка` |
| Полы | `OST_Floors` | `Группа модели` | `Пол` |
| Потолки | `OST_Floors` | `Группа модели` | `Потолки` |

- Название классифицирующего параметра в V1 фиксировано.
- Ожидаемые значения редактируются; UI предварительно предлагает обнаруженные значения.
- `OST_Ceilings` в V1 не подключается.
- Описание берётся из выбранного текстового параметра типа стен и перекрытий, по умолчанию `Состав`.
- Отсутствие выбранного параметра у части типов блокирует запуск и показывает проблемные типы.

### Входные помещения и область

- `По этажу`: только валидные помещения выбранного уровня.
- `По секции или корпусу`: фильтр по выбранному параметру экземпляра помещения и его значению.
- `По всему объекту`: все валидные помещения выбранной фазы.
- Неразмещённые, незамкнутые и избыточные помещения пропускаются с предупреждением.
- Идентификатор помещения: системные `Номер`/`Имя` через `BuiltInParameter` либо совместимый пользовательский текстовый параметр экземпляра.

### Геометрия и количество

- Один `SpatialElementGeometryCalculator` создаётся на запуск с `SpatialElementBoundaryLocation.Finish`.
- Стены сначала определяются через `SpatialElementBoundarySubface`; площадь берётся из относящейся к помещению subface.
- Для стены, не ставшей прямой границей, разрешён только проверяемый резервный алгоритм по кандидатам, плоскостям, нормалям и площади пересечения. Полную площадь стены брать нельзя.
- Полы и потолки определяются пересечением solid перекрытия с тонкими probe-solid у нижней и верхней границ room solid.
- Один элемент, проходящий через несколько помещений, порождает отдельный `FinishOccurrence` с площадью части для каждого помещения.
- В V1 допустимы горизонтальные перекрытия без сложного shape editing. Наклонная или неоднозначная геометрия даёт предупреждение, а не приближённый тихий результат.
- Revit API не вызывается через `Task.Run` или `Parallel.ForEach`.

### Агрегация

- В ключ группы входят нормализованные, естественно отсортированные описания только включённых категорий.
- Площадь в ключ группы не входит.
- Одинаковые описания внутри помещения и затем внутри группы суммируются численно.
- Округление и преобразование площади в текст выполняются только на финальном форматировании.
- Порядок строк описаний и площадей всегда одинаков; количество строк совпадает.
- `NoFinish` выводится как `—`; `Unknown` сопровождается предупреждением и не превращается в обычный прочерк.
- Пустое описание элемента отображается как `[Описание не задано]` и создаёт предупреждение.

### Запись и спецификация

- В каждое помещение группы записывается один и тот же список помещений и одинаковые агрегированные значения.
- Площадь хранится текстом с двумя знаками после запятой, без `м²` в значении.
- Отключённая категория не участвует в ключе, не требует выходных параметров и не добавляется в спецификацию.
- Отсутствующие общие параметры V1 не создаёт автоматически.
- Опциональная запись принадлежности в физические элементы хранит реальные идентификаторы помещений, которых касается элемент; она не используется как источник последующего расчёта.
- Изменения выполняются через `TransactionGroup`: ownership, параметры помещений, спецификация, metadata.
- Ошибка обязательной записи хотя бы одного целевого помещения откатывает результат, чтобы не создавать частично корректную ведомость.
- Спецификация помечается Extensible Storage схемой `TrueBIM.FinishSchedule`. Чужая спецификация с тем же именем не изменяется и не удаляется.

## Ограничение агрегированных профилей

Один комплект выходных параметров помещений хранит один активный вариант группировки. Новый запуск с другим scope может перезаписать прежние агрегированные значения.

Перед выполнением UI показывает предупреждение:

> Выбранные параметры помещений будут обновлены. Один комплект параметров поддерживает один активный вариант агрегации.

Несколько одновременных output profiles и служебные `ReportRow` остаются будущим развитием.

## Целевая архитектура

```text
FinishScheduleCommand
└── FinishScheduleWindow
    ├── FinishScheduleSettingsService
    ├── ParameterCatalogService
    └── FinishScheduleWorkflow
        ├── RoomScopeService
        ├── FinishElementCollector
        ├── IFinishQuantitySource
        │   └── PhysicalFinishQuantitySource
        ├── RoomFinishSnapshotBuilder
        ├── FinishAggregationService
        ├── FinishOwnershipWriter
        ├── RoomFinishParameterWriter
        └── FinishRoomScheduleBuilder
```

Правила границ:

- `UI` не содержит collectors, геометрические операции и транзакции.
- `Models` не зависят от WPF и по возможности не зависят от Revit API.
- `Revit` содержит чтение документа, geometry, writers и schedule builder.
- `Services` содержит чистую нормализацию, группировку, форматирование и orchestration.
- `IFinishQuantitySource` вводится до геометрической реализации. В V1 существует только `PhysicalFinishQuantitySource`; фиктивный `AssumedFinishQuantitySource` не создаётся.
- Строковые имена параметров не проникают в чистую агрегацию: она получает `ParameterReference` и готовые снимки значений.

### `FS-002` — модель настроек и каталог параметров

Состояние: выполнен в коде, runtime smoke каталога ожидает Revit.

- Добавить `FinishScheduleSettings`, `FinishCategorySettings`, `RoomIdentifierSettings`, `ReportScopeSettings`, `ParameterReference`.
- Реализовать `ParameterCatalogService` с различением instance/type, категории, `StorageType`, writable/read-only, системного/project/shared parameter, GUID и definition `ElementId`.
- Ввести стабильную identity параметра без зависимости только от локализованного имени.
- Добавить чистую валидацию обязательных полей и совместимости категорий.
- Покрыть defaults, identity и validation unit-тестами.

Критерий готовности: сервис возвращает только совместимые варианты и объясняет каждую причину блокировки.

Реализовано:

- immutable-модели содержат безопасные defaults из ТЗ и не зависят от Revit API;
- built-in identity опирается на `BuiltInParameter`, shared identity — на GUID, project identity — на definition `ElementId`; локализованное имя используется только для отображения;
- read-only `ParameterCatalogService` собирает экземпляры и уникальные типы помещений, стен и перекрытий без транзакций;
- каталог фиксирует binding, `StorageType`, применимые категории и фактическую доступность записи на проверенных элементах;
- чистый matcher возвращает все причины несовместимости, а validator защищает обязательные поля, отключённые категории, scope, дубли и source/output conflicts;
- defaults, identity, фильтрация совместимых вариантов и validation покрыты unit-тестами.

### `FS-003` — полное окно и сохранение профиля

Состояние: выполнен в коде, визуальный runtime smoke ожидает Revit.

- Реализовать восемь блоков одного окна: классификация, описание, идентификатор, ownership, категории, выходные параметры, scope, имя спецификации.
- Использовать редактируемые ComboBox для значений классификации.
- Отключать зависимые поля при выключенной категории или ownership.
- Сохранять профиль через адаптер над `JsonSettingsStorage` с ключом `finish-schedule`.
- Не включать `Сформировать`, пока настройки не валидны.

Критерий готовности: несовместимая конфигурация не запускается, а сохранённый профиль безопасно восстанавливается.

Реализовано:

- окно содержит восемь последовательных блоков без вкладок и сохраняет рабочую плотность через единый вертикальный scroll;
- значения классификации редактируются, а зависимые поля категорий, ownership, пользовательского идентификатора и scope отключаются вместе с родительским выбором;
- dropdown-параметры строятся только из совместимых вариантов каталога; description динамически учитывает все включённые физические категории;
- footer и validation banner сразу показывают причины блокировки, включая обязательные поля, read-only, дубли и source/output conflicts;
- профиль сохраняется адаптером над `JsonSettingsStorage` с ключом `finish-schedule`; built-in, shared и project identities round-trip без привязки только к имени;
- повреждённый JSON возвращает безопасные defaults, а некорректная сохранённая ссылка отбрасывается без потери остальных полей;
- `Сформировать` требует одновременно валидную конфигурацию и подключённый workflow, поэтому FS-003 по-прежнему не открывает транзакции и не меняет модель.

### `FS-004` — scope и сбор кандидатов

- Реализованы `RoomScopeService`, `FinishElementCollector`, `FinishClassificationService` и orchestration-сервис preview.

- Значение `Группа модели` хранится в кэше типов, а выбранный параметр секции снимается в snapshot помещения с сохранением raw/display значения.
- Предварительный `FinishBoundingBoxIndex` отсортирован по `MinX` и сокращает число проверяемых пар помещение–элемент без вычисления точной геометрии.
- Read-only preview показывает валидные, невалидные и отфильтрованные помещения, source/classified/in-scope количества стен, полов и потолков, элементы без bounds и потенциальные пары.
- Любое изменение влияющих настроек помечает preview устаревшим; повторный сбор выполняется только по явной кнопке и не открывает транзакцию.

Критерий готовности выполнен в коде и unit-тестах: результаты уровня, секции и всего объекта детерминированы и не требуют полного collector на каждое помещение. Runtime-сверка с реальной моделью остаётся отдельной проверкой.

### `FS-005` — физический источник количества и геометрия

- Добавлены чистый контракт `IFinishQuantitySource`, `PhysicalFinishQuantitySource`, числовой `FinishOccurrence` и детерминированный accumulator по ключу room/element/category.
- Один `SpatialElementGeometryCalculator` на запуск обслуживает `FinishRoomGeometryCache`; room solid и boundary results повторно не вычисляются.
- Прямые стены получают площадь только из относящихся к помещению `SpatialElementBoundarySubface` типа `Side`.
- Резервный алгоритм стен пересекает solid элемента с двусторонним тонким probe каждой вертикальной грани и выбирает параллельную контактную грань; полная площадь стены не используется.
- Полы и потолки пересекаются с независимыми probes нижних и верхних горизонтальных граней room solid; площадь берётся как одна максимальная горизонтальная грань каждого intersection solid, без суммы верхней и нижней поверхности.
- Одна плита формирует отдельный occurrence для каждого помещения, поэтому площадь распределяется по фактическому overlap, а не копируется целиком.
- Наклонная/неоднозначная геометрия, ошибки boolean intersection, отсутствие solid и недоказанный fallback превращаются в типизированные warnings и не дают тихую приближённую площадь.
- Read-only preview показывает число геометрических связей и суммарную площадь по категориям; транзакции по-прежнему не открываются.

Критерий готовности выполнен в архитектуре и unit-тестах: одна плита хранит независимые occurrences по помещениям, а верхняя и нижняя горизонтальные грани одного пересечения не удваиваются. Runtime-сверка площадей на реальной модели остаётся отдельной проверкой.

### `FS-006` — чистая нормализация и агрегация

- Реализованы `RoomFinishSnapshotBuilder`, `FinishDescriptionNormalizer`, `FinishGroupKeyBuilder`, `FinishAggregationService` и отдельный финальный formatter.
- Описания нормализуются через `Trim`, схлопывание whitespace/newlines и регистронезависимый comparison key; пустое значение получает `[Описание не задано]` и warning.
- Ключ группы содержит только состояния и естественно отсортированные описания включённых категорий; площади и отключённые категории на группировку не влияют.
- Повторы описаний суммируются числами сначала внутри помещения, затем внутри группы; округление до двух знаков выполняется только при создании текстового output.
- Идентификаторы помещений и строки отделки сортируются естественно, а description/area формируются в одинаковом порядке с одинаковым числом строк.
- `NoFinish` форматируется как `—`; `Unknown` остаётся отдельным состоянием с явным маркером и warning.
- Один immutable output группы переиспользуется для всех входящих помещений, а read-only preview показывает количество групп и подготовленных помещений.

Критерий готовности выполнен в коде и 15 обязательных unit-тестах: группировка и форматирование детерминированы независимо от порядка входных элементов. Runtime-сверка на реальной модели остаётся отдельной проверкой.

### `FS-007` — запись ownership и параметров помещений

- Реализованы чистые builders целевых значений, `FinishParameterChangePlanner`, `FinishOwnershipWriter`, `RoomFinishParameterWriter` и `FinishScheduleWriteWorkflow`.
- Geometry, scope, quantity и aggregation полностью выполняются до начала `TransactionGroup`; ownership ограничен bounding-box кандидатами текущего scope.
- Preflight повторно находит параметр по стабильной identity и проверяет каждый target element, `StorageType.String`, read-only и worksharing checkout status.
- План хранит старые и новые значения, отделяет unchanged от реальных изменений и показывается пользователю до подтверждения записи.
- Одинаковые значения группы записываются во все целевые помещения; повторный запуск с актуальными значениями не открывает транзакции.
- Обязательные Room-параметры записываются отдельной транзакцией внутри `TransactionGroup`; любой отказ записи откатывает также ранее выполненный ownership-срез.
- Ownership использует реальные идентификаторы помещений, natural sort и удаление дублей; locked/missing элементы пропускаются с warnings.
- Элемент без помещений текущего scope получает план очистки с warning, но `Unknown` geometry не очищает ранее записанную принадлежность.
- UI включает `Сформировать` только для валидной конфигурации, показывает примеры old/new и предупреждает об одном активном aggregation profile.

Критерий готовности выполнен в коде и unit-тестах: повторный запуск идемпотентен, обязательный blocker запрещает старт записи, а optional ownership blocker остаётся warning. Runtime-проверка rollback и worksharing locks остаётся отдельной ручной проверкой.

### `FS-008` — создание и обновление спецификации

- Реализованы `FinishRoomScheduleBuilder`, чистый `FinishRoomSchedulePlanBuilder` и `FinishScheduleMetadataService`.
- Поля добавляются только для включённых категорий в порядке `помещения → стены → потолок → пол`; заголовки и ширины соответствуют `40/80/25 мм`.
- `IsItemized` отключён, сортировка выполняется по перечню помещений; для уровня и раздела создаётся скрытое поле с typed equality filter.
- Extensible Storage хранит marker `TrueBIM.FinishSchedule`, версию schema, settings hash, scope, stable parameter identities и UTC последнего обновления.
- Повторный запуск находит управляемую спецификацию независимо от переименования настройки, обновляет её на месте и не создаёт дубль; неизменная конфигурация не открывает транзакцию спецификации.
- Чужая спецификация с тем же именем и несколько ведомостей с marker TrueBIM блокируют весь атомарный workflow до начала записи.
- Создание/обновление спецификации выполняется третьей транзакцией в том же `TransactionGroup`, что ownership и Room-параметры; ошибка откатывает весь результат.
- Версионная конвертация ширин из миллиметров изолирована в adapter для legacy `DisplayUnitType` и нового `UnitTypeId`.

Критерий готовности выполнен в коде и unit-тестах чистого плана: повторный запуск обновляет только собственную Room Schedule и не изменяет пользовательские виды. Runtime smoke на реальной модели остаётся отдельной ручной проверкой.

### `FS-009` — производительность, отчёт и отказоустойчивость

## Обязательные unit-тесты чистой логики

1. Одинаковые описания и разные площади дают одну группу.
2. Разные описания дают разные группы.
3. Отключённая категория не влияет на ключ.
4. Реальное отсутствие отделки даёт `—`.
5. `Unknown` не превращается в `—` без предупреждения.
6. Несколько описаний дают синхронные многострочные значения.
7. Дубли описаний внутри помещения суммируются.
8. Дубли описаний между помещениями группы суммируются.
9. Площадь не входит в ключ.
10. Идентификаторы помещений сортируются естественно.
11. Пробелы, переносы и регистр нормализуются.
12. Результат не зависит от порядка входных элементов.
13. Округление выполняется только при выводе.
14. Пустое описание создаёт предупреждение.
15. Каждое помещение группы получает идентичный output.

## Обязательные manual/integration сценарии

- одинаковый пол при разных площадях помещений;
- разные полы;
- два вида пола или стен внутри помещения;
- отсутствующий потолок;
- одно перекрытие через несколько помещений;
- элемент без помещения;
- пустое описание;
- неразмещённая и незамкнутая комната;
- scope по уровню, секции и всему объекту;
- повторный запуск;
- чужая schedule с тем же именем;
- read-only/locked параметры и worksharing elements;
- отключённая ownership-запись и отключённые категории.
